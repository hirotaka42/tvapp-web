本書は TVapp（TVER 視聴 Web アプリ）に対して、どの単位で・何を・どの入力で確かめるかという テスト観点を網羅的に定義する。Cloudflare + Firebase への $0 移行に伴い、これまで Azure Functions + yt-dlp が 担っていたストリーム解決を 純 TypeScript の API ルート内実装へ差し替える。この差し替えは TDD（テスト先行）で進める。 本書はその設計対象と、既存 TVER 代理ルート・認証・UI の観点を、実コードとデータフローの事実に沿って表で整理する。

本書は TDD（テスト先行）で実装予定の純 TS ストリーム解決器に対するテスト観点を定義する。 現状 package.json にテストフレームワークは未導入（scripts は dev / build / start / lint のみ）で、 vitest はこの移行の一部として導入する。ストリーム解決の記述対象は移行後の純 TS 解決器であり、 既存実装は参考に留める（現行は Azure 依存の src/app/api/service/call/streaminglink/route.ts）。差し替え前後で観点が変わる箇所は明記する。

2. テスト戦略

2.1 TDD（テスト先行）の運用

純TSストリーム解決は「TVER の実レスポンスを固定化したフィクスチャ」に対して先にテストを書き、 Red → Green → Refactor で実装する。外部 API はネットワークに出さず、fetch をモックして 記録済みレスポンス（正常・各種異常）を差し込む。これにより geo 制限や月替わりキーなど、 実行環境・時刻に依存する条件を決定論的に再現する。

  1. Red: 観点（例「streaks 無しの旧作は Brightcove へフォールバックする」）を表明する失敗テストを書く。
  2. Green: 最小実装で通す。
  3. Refactor: 重複除去・境界の明確化。テストは緑のまま。

2.2 テストの階層（vitest）

階層粒度対象外部依存の扱い
ユニット純関数月替わりキー選択、キー key0{idx} 算出、sources から非DRM抽出、version 抽出、URL 組み立て依存なし（引数のみ）。時刻はテストから注入
統合（結線）解決フロー全体session→callEpisode→statics→streaks_info→playback を通した m3u8 解決、各段の失敗伝播fetch をモックし段ごとに応答を差し込む
ルート（API）Route Handler各 route.ts の 400/上流ステータス伝播/500、middleware の 401NextRequest を組み立て、fetch をモック
E2E（実ブラウザでの再生・実 TVER 疎通）は本仕様の対象外。geo（JP限定）・月替わりキーは実行時刻/回線に依存するため、 自動テストはフィクスチャと注入した時刻で決定論的に検証し、実疎通は手動確認に委ねる。

2.3 境界値・異常系の基本方針

  • 必須パラメータ: 欠落 / 空文字 / 空白のみ を境界として区別する（既存の代理ルートには空白のみも弾く実装があり、純 TS 解決器でも同じ方針で弾く。§5.1 参照）。
  • 月替わりキー: JST の全 12 か月 + 6月/12月の % 6 === 0 分岐を境界として全数検証する（§4）。
  • 上流ステータス: 200 以外（403 / 404 / 5xx）をそのままクライアントへ伝播することを確認する。
  • DRM 除外: key_systems を持つ source を除外し、非DRMが 0 件のときの挙動を境界として確認する。

3. ストリーム解決のテスト観点

純TS解決器は、エピソードID を起点に TVER の公開 API を順に辿り、非DRMのHLS(m3u8)を得る。 全リクエストは Origin / Referer を TVER 相当に固定した素の HTTP で、認証は各段のトークン/キーで行う。 月替わりキー選択と Origin: https://tver.jp が要になる。

下図は純TS解決器がサーバ側で辿る上流 TVER API の5段（/callEpisode などは上流 TVER のパス短縮表記）。 フロントが叩く BFF ルート（/api/service/*）は §5 で別途扱う。 ランキング（callEpisodeRankingDetail）によるホーム合成は解決フローとは別系統で、§5 の代理ルートに含める。

3.2 純関数として切り出す観点（ユニット）

フローから独立してテストしやすい純関数を、純 TS 解決器で新たに切り出して実装する（現状未実装・TDD で導入する設計）。境界値をユニットで固める。以下の関数名は実装予定の設計上の呼称。

5. TVER 代理ルートのテスト観点

BFF の各 Route Handler は、必須クエリを検証してから TVER 公開 API を叩き、上流の失敗をステータスごと伝播する。 実装は概ね「必須欠落→400」「!response.ok→上流の response.status を返す」「例外→500」で統一されている。

5.1 必須パラメータと伝播

ルート必須欠落時上流失敗時成功時
POST /api/service/session—（body 固定 device_type=pc）—上流ステータス伝播（例 5xx）{ platformUid, platformToken }
GET /api/service/call/episodeepisodeId / platformUid / platformToken400「必須です」上流ステータス伝播{ data }
GET /api/service/call/ranking/episode/detail/[genre]genre（空白のみも弾く）400上流ステータス伝播{ data }
GET /api/service/call/homeplatformUid / platformToken400各 genre 個別失敗は null で除外し合成継続ranking を callHome 形へ合成
GET /api/service/call/seriesEpisodes/[seriesId]seriesId / platform_uid / platform_token400上流ステータス伝播{ data }
GET /api/service/searchkeyword / platformUid / platformToken400上流ステータス伝播{ data }
GET /api/content/episode/[episodeId]episodeId400上流ステータス伝播{ data }（statics episode.json）
GET /api/service/call/streaminglinkepisodeId（空白のみも弾く）400上流ステータス伝播（現行 Azure 依存 → 純TSへ差替）{ video_url }

5.2 代理ルート共通の観点

  • ヘッダ固定: TVER 系は x-tver-platform-type: web ・Origin/Referer=tver.jp（session のみ s.tver.jp）。リクエスト検証でこれらが付くことを確認する。
  • ステータス伝播: 上流 403/404/5xx を response.status のまま返し、500 に丸めないことを確認する。
  • 例外時 500: fetch 自体が throw（ネットワーク断など）したときのみ 500 を返す。
  • home の部分失敗耐性: 5ジャンル中いくつかが失敗しても、成功分だけで合成を続ける（filter(r => r !== null)）。全滅時の挙動も観点化する。
  • クエリのエスケープ: search の keyword が URL に素で連結される点。特殊文字（空白・記号・日本語）を境界として検証する。
// 例: 必須欠落 → 400
it('episodeId が無ければ 400', async () => {
  const req = new NextRequest('http://x/api/service/call/episode');
  const res = await GET(req);
  expect(res.status).toBe(400);
});
// 例: 上流 403 をそのまま伝播
it('上流 403 を 403 で返す', async () => {
  vi.spyOn(globalThis, 'fetch').mockResolvedValue(new Response('', { status: 403 }));
  const req = new NextRequest('http://x/api/service/call/episode?episodeId=a&platformUid=u&platformToken=t');
  const res = await GET(req);
  expect(res.status).toBe(403);
});

6. 認証のテスト観点

認証は 2 層。middleware が /api/ 配下の Authorization: Bearer 有無を判定し、 無ければ 401 を返す（公開ルートは除外）。各 API は Bearer トークンを検証し、匿名（ゲスト）はロールと視聴制限で扱いを変える。

6.1 middleware（トークン有無）

入力期待
/api/* かつ Authorization 無し（または Bearer で始まらない）401「トークンがありません。」
公開ルート（/api/User/Register, /api/User/setGuestRole, /api/utils/verify-token, /api/health 等）トークン無しでも next() で通過
非 /api/ パス（ページ）でトークン無し401 にせず通過（詳細は各画面/ API で判定）
/about / /anotherredirect / rewrite（レガシールート）
middleware はトークンの中身を検証しない（存在のみ判定）。中身の検証は各 API（verify-token の jwtVerify、または firebase-admin の verifyIdToken）が担う。この責務分担を観点として明示する。

6.2 トークン検証（各 API）

対象観点期待
POST /api/utils/verify-token有効な JWT（JWT_SECRET_KEY 署名）{ valid: true, payload }
POST /api/utils/verify-token無効／改竄／期限切れ JWTverifyToken が null → 401 { valid: false }
POST /api/utils/verify-tokenJWT_SECRET_KEY 未設定verifyToken が throw → 500
firebase-admin 系（favorites 等）(段階2)Authorization 無し401「認証が必要です」
firebase-admin 系不正トークン（auth/argument-error）401「トークンが無効です」
firebase-admin 系本文バリデーション（seriesId/seriesTitle 空）400

6.3 匿名ゲストの制限

対象観点期待
POST /api/User/setGuestRole有効な匿名トークンCustom Claims に role=GUEST を設定し、ゲストプロファイルを作成
POST /api/User/setGuestRoleAuthorization 無し401「認証が必要です」
エピソード再生（ゲスト）loginUser.isAnonymous かつ playedSeconds >= 30再生停止し 30秒制限モーダルを表示、10秒カウントダウン後 / へ遷移
エピソード初回アクセス（ゲスト）episode 取得後の初回視聴時間制限の初回警告モーダルを表示

7. UI のテスト観点

エピソードページ（src/app/episode/[episodeId]/page.tsx）は、認証状態・データ取得・タイムアウトで 表示を切り替える。状態遷移を観点として固定する。

状態条件表示
認証待ち!loginUsernull（何も描画しない）
episodeId 欠落!episodeIdErrorState「エピソードIDが指定されていません」＋ホームリンク
データ取得中!episode かつ 未タイムアウトnull（＋動画枠は「動画を読み込み中...」）
タイムアウト!episode かつ 10秒経過（loadingTimeout）ErrorState「エピソードが見つかりません」＋再読み込み(onRetry)
動画URL未取得episode あり / videoUrl なし「動画を読み込み中...」プレースホルダ
再生可能videoUrl.video_url ありVideoPlayer（react-player, HLS）を描画
ゲスト初回isAnonymous ＋ episode 取得初回警告モーダル（「了解しました」）
ゲスト制限到達30秒到達制限モーダル（ボタン無し・自動遷移カウントダウン）
タイムアウト判定は setTimeout(..., 10000) に依存する。テストでは vi.useFakeTimers() で 10秒を進め、episode 未取得のままタイムアウト表示に切り替わることを検証する（実時間を待たない）。
  • ローディング: 初期は null／プレースホルダ。episode と videoUrl の到着で描画が進むこと。
  • タイムアウト: 10秒経過かつ未取得で ErrorState。到着済みならタイムアウト表示に落ちないこと（境界: 到着が9.9秒 vs 10.1秒）。
  • エラー表示: episodeId 欠落とタイムアウトで異なる文言・アイコン。再読み込み導線（onRetry / window.location.reload）の存在。
  • モーダル排他: モーダル表示中は isPlaying=false に強制され、controls も無効化されること。
