TVapp CI/CD アーキテクチャ(版ゲート自動デプロイ)

対象 tvapp-web(xiaovy.tvapp.web/) ブランチ project/tvapp-cf-zero-cost ステータス 実装済み(運用中)
関連 .github/workflows/ci.yml 関連 .github/workflows/deploy.yml 関連 xiaovy.tvapp.web/wrangler.jsonc

1. 概要

TVapp のデプロイは GitHub Actions による版ゲート方式 CI/CD で構成する。 リポジトリは hirotaka42/tvapp-web、Next.js アプリ本体はリポジトリ直下ではなく xiaovy.tvapp.web/ サブディレクトリに置かれている。したがって各ワークフローの ジョブは working-directory: xiaovy.tvapp.web を既定として動作する。

ワークフローは 2 本に分かれる。品質を守る CI(ci.yml)と、版が上がったときだけ本番へ出す 版ゲート・デプロイ(deploy.yml)である。デプロイ先は Cloudflare Workers で、 Next.js を OpenNext でバンドルし wrangler で配信する。

版ゲート(AGENTS-SHARED §4):「版が上がったときだけ」出荷する。 xiaovy.tvapp.web/package.json の version を上げない変更は、 main に入ってもデプロイされない。

2. 全体フロー

3. CI(ci.yml)— 品質ゲート

壊れた変更を main に入れないためのゲート。Pull Request と、main 以外への push (branches-ignore: [main])で起動する。同一 ref の実行は直列化され、後発が来ると 古い実行はキャンセルされる(concurrency: ci-${{ github.ref }} / cancel-in-progress: true)。

verify ジョブは ubuntu-latest 上、working-directory: xiaovy.tvapp.web、 Node 20(npm キャッシュ、cache-dependency-path: xiaovy.tvapp.web/package-lock.json)で以下を順に実行する。

  1. 依存インストール — npm ci
  2. 型検査 — npx tsc --noEmit
  3. Lint — npm run lint
  4. テスト(ユニット) — npm test
  5. 本番ビルド(Next) — npm run build
  6. OpenNext ビルド(Workers バンドル生成確認) — npm run cf:build

ビルド検証用に NEXT_PUBLIC_* 系の値を GitHub の Variables から env に注入するが、 これらは実行時のみ必要な公開値のため未設定でもビルド自体は通る。

この CI はゲートとして機能させることを前提とする。ブランチ保護で verify を 必須チェックに設定し、緑でなければ main にマージできない運用を推奨する。

4. 版ゲート・デプロイ(deploy.yml)

main への push(= PR マージ)時に起動する(on: push: branches: [main])。 concurrency: deploy-main / cancel-in-progress: false で、main への多重デプロイを 直列化する(実行中のデプロイはキャンセルしない)。版タグ付与のため permissions: contents: write を持つ。 ジョブは gate と deploy の 2 段。

4-1. gate ジョブ — 版変化の判定

fetch-depth: 2 で直前コミットまで取得し、xiaovy.tvapp.web/package.json の version を比較する。

  • 現在(CUR)= 現在の package.json の version。
  • 直前(PREV)= git show HEAD~1:xiaovy.tvapp.web/package.json の version(取得不可なら none)。
  • CUR != PREV なら deploy=true、同じなら deploy=false を出力(version も出力)。

つまり version を上げたコミットが main に入ったときだけ後続のデプロイが走る。

4-2. deploy ジョブ — 本体(needs: gate / if: deploy == 'true')

working-directory: xiaovy.tvapp.web、Node 20(npm キャッシュ)で以下を順に実行する。 ビルド時に埋め込む NEXT_PUBLIC_* は Variables から注入する。

  1. 依存インストール — npm ci
  2. テスト(ユニット) — npm test
  3. OpenNext ビルド — npm run cf:build
  4. Cloudflare Workers へデプロイ — npx wrangler deploy(CLOUDFLARE_API_TOKEN / CLOUDFLARE_ACCOUNT_ID を Secrets から注入)
  5. 版タグを付与 — v{version} を作成して push(同名タグが既存ならスキップ)

デプロイ対象の Worker 設定は xiaovy.tvapp.web/wrangler.jsonc に定義する: Worker 名 tvapp-web、エントリ .open-next/worker.js、 compatibility_date: 2025-03-25、compatibility_flags: ["nodejs_compat"]、 静的アセット .open-next/assets(バインディング ASSETS)、observability 有効。

5. 必要な GitHub 設定

CI とデプロイを機能させるために、リポジトリに次を設定する。

5-1. Secrets(機密・デプロイ本体で使用)

名前用途
CLOUDFLARE_API_TOKENCloudflare Workers を編集する権限を持つ API トークン。wrangler deploy の認証に使用。
CLOUDFLARE_ACCOUNT_IDデプロイ先の Cloudflare アカウント ID。

5-2. Variables(公開値・ビルド時に埋め込み)

NEXT_PUBLIC_* は ビルド時にクライアント側バンドルへ埋め込まれるため、 CI / デプロイのビルド環境に必要。CI・deploy 両ワークフローが Variables から env へ注入する。

名前用途
NEXT_PUBLIC_FIREBASE_API_KEYFirebase 公開 API キー。
NEXT_PUBLIC_FIREBASE_AUTH_DOMAINFirebase 認証ドメイン。
NEXT_PUBLIC_FIREBASE_PROJECT_IDFirebase プロジェクト ID。
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKETFirebase ストレージバケット。
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_IDFirebase メッセージング送信者 ID。
NEXT_PUBLIC_FIREBASE_APP_IDFirebase アプリ ID。
NEXT_PUBLIC_IDTOKEN_NAMEID トークンの識別名(クライアントで参照)。

5-3. Actions / ワークフローの権限

項目内容
Actions の書き込み権限リポジトリの Actions 設定で workflow の書き込みを許可する(版タグ付与に contents: write を使用)。
workflow ファイルの pushワークフローファイル(.github/workflows/*)を push するには、トークンに workflow スコープが必要(SSH 鍵でも可)。

6. 運用フロー

  1. feature ブランチで開発する。
  2. PR を作成し CI(ci.yml)が緑になることを確認する。
  3. レビューを受ける。
  4. 出荷するなら xiaovy.tvapp.web/package.json の version を上げて main にマージする。
  5. deploy.yml が版上げを検知し、Cloudflare Workers へ自動デプロイする。
  6. デプロイ成功後、v x.y.z タグが付与される。
version を据え置いた場合(挙動が同じリファクタなど)は、main にマージしても gate が deploy=false と判定し、デプロイは走らない。

7. 補足

  • Cloudflare の無料枠で運用する構成(Workers + OpenNext + Wrangler)。
  • TVER は日本国内限定で配信される(事実)。
  • ブランチ保護の推奨: main への直 push を禁止し、ci.yml の verify を 必須チェックに設定する。PR 経由でのみ main を更新し、CI 緑 + レビュー通過を取り込みの条件とする。
