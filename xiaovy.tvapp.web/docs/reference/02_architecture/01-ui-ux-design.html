TVapp UI/UX 設計書

対象=tvapp-web（xiaovy.tvapp.web / Next.js 14.2 App Router） ブランチ=project/tvapp-cf-zero-cost 関連=Main.tsx / episode/[episodeId] / atomicDesign ステータス=現状記述 + 改善提案（WIP）

本書は TVER 視聴 Web アプリ「TVapp」フロントエンドの UI/UX 設計を、実装コード （src/app / src/components/atomicDesign）に即して記述する。画面一覧・遷移、情報設計、 コンポーネント階層（atomic design）、テーマ（next-themes）、レスポンシブ、アクセシビリティ、 ゲスト視聴の 30 秒制限 UX、動画再生（react-player）を扱う。末尾に、コードから観測できる事実に基づく 改善提案を客観的に併記する。記述は技術・構成・データフローの事実に徹し、題材の是非は評価しない。

本アプリの UI は Next.js App Router のサーバ/クライアント境界上に構築される。 描画コンポーネントの大半は "use client" で、データは Context 経由で注入した Service 群 （SessionService / TvHomeService / EpisodeService / StreamService ほか） が同アプリの BFF（/api/*）を叩いて取得する。UI は取得済みデータを描くことに集中する。

1. 全体像とレイアウトシェル

ルートは src/app/layout.tsx。<html lang="ja" suppressHydrationWarning> の下に Providers（next-themes + 認証 + 各 Service Context + トースト）でラップし、常設の Header と各ページ（children）を配置する。テーマの初期不一致（ハイドレーション警告）を 避けるため suppressHydrationWarning を付与している。body には min-h-screen dark:bg-black dark:text-white を与え、ダーク時の地色を黒基調にする。

Header は usePathname() を見て /user/login と /user/register では null を返し、認証前画面ではシェル（ハンバーガー）を出さない。ログイン画面は左半分の ブランドパネル（BrandPanel）と右半分のフォームという独立レイアウトを持つ。

3. 画面遷移

未認証時の入口は必ずログイン画面。ホーム（Main）は useAuth() と useFirebaseAuth().loading を監視し、!loading && !loginUser のとき router.push('/user/login') でリダイレクトする。ゲスト（匿名）は閲覧できるが、 エピソードでの視聴は 30 秒で制限され、達成後はホームへ自動遷移する（§8）。

図1 の凡例とテキスト説明（スクリーンリーダ向け）
  • 青枠＝ゲストでも閲覧できる画面（/user/login・/ホーム・/episode/[episodeId]）／灰枠＝ログイン（非ゲスト）が要る画面（/user/register・/series/[seriesId]・/user/profile 系・/admin/users）。
  • 実線矢印＝ユーザー操作による遷移／青の破線矢印＝ゲストの 30 秒制限による自動遷移。

遷移の流れ: /user/login（ログイン / ゲスト）で認証に成功するとホーム / へ進む。/user/register（新規登録）はログイン画面と相互に行き来する。ホームのカードを選ぶとエピソード /episode/[episodeId] へ、シリーズ名からシリーズ一覧 /series/[seriesId] へ、シリーズ一覧の各エピソードからエピソードへ遷移する。ホームのヘッダ内サイドパネルからプロフィール（/user/profile・favorite・history）へ、role===99 のときのみユーザー管理 /admin/users へ入る。ゲストがエピソードで再生 30 秒に達すると、破線の経路でホーム / へ自動的に戻る。

4. 情報設計（画面別）

本節では、ユーザーが頻繁に触れるホーム・エピソード・ログインの 3 画面を詳述する。 その他のページ（シリーズ一覧・プロフィール・お気に入り・視聴履歴・ユーザー管理・ライブ再生）は §5 のコンポーネント（GenreContentCardList・ProfileCard・各フォーム等）の組版で構成でき、 情報構造が上記 3 画面の応用に収まるため、本節では図示を省略する。

4.1 ホーム（Main.tsx）

ホームは「ジャンル/ラベル別ランキングの縦積み × 各行の横スクロール」で構成する。 useTvHomeService(session) の結果から getAllLabels() でラベル配列を得て、 ラベルごとに getContentsByLabel() → convertRankingToCardData() でカード配列へ整形し、 状態 rankingContents / rankingLabels に格納する。描画は rankingLabels.map(label => <h2>{label}</h2> + <ContentCardList>)。

横スクロール行の実体は ContentCardList の flex flex-row ... gap-x-4 overflow-x-auto。各カード ContentCard は 16:9 サムネイル（aspect-w-16 aspect-h-9）、任意のランクバッジ（rank があれば https://tver.jp/images/PC_img_ranking_{rank}.svg を左上に重畳）、シリーズ名 / エピソード名 / 「制作局 放送日ラベル」を truncate で 1 行省略表示する。カードは /episode/{id} へのリンク。

ホーム合成の背景（データ層の事実）: TVER の callHome は空を返すため、UI は callEpisodeRankingDetail/{genre}（drama / variety / anime / documentary / sports）の ランキングを合成してホームを構成する。ラベル分割・カード整形は utils/Convert/ranking/home/responseParser の getAllLabels() / getContentsByLabel() と convertRankingToCardData が担う。

4.2 エピソード視聴（episode/[episodeId]/page.tsx）

視聴画面は上から「16:9 プレイヤー」「メタ情報ブロック」「シリーズ内エピソード一覧」の縦積み。 プレイヤーの寸法は width:100vw; height:calc(100vw * 9 / 16) の固定 16:9 で、背景は黒 （#000）。videoUrl 取得前は「動画を読み込み中...」を中央表示する。 メタブロック（width:95vw）はシリーズ名リンク（/series/{seriesID}）+ お気に入りボタン （非ゲストのみ）、<h3> のエピソード名、放送局ラベル・放送日ラベル・説明、TVER の URL（https://tver.jp/episodes/{episodeId} へのリンク）を並べる。

シリーズ内一覧は useSeriesService(seriesID, session) の結果を convertCardContentsBySeason() でシーズン別にまとめ、シーズンごとに見出し + GenreContentCardList（レスポンシブグリッド）で表示する。エピソード情報が 10 秒以上取得できない場合 （!episode の状態が続く場合。setTimeout による判定）は ErrorState（「エピソードが見つかりません」+ 再読み込み/ホーム）へ切り替える。

4.3 ログイン / ゲスト（user/login/page.tsx）

ログインは min-h-screen flex の 2 カラム。左は BrandPanel （hidden lg:flex lg:w-1/2、インディゴ→ブルーのグラデーションにロゴ・キャッチ・特徴 3 点・ TV イラストの SVG）で、lg 未満では非表示。右は SignInForms（カード内にロゴ・ 「ログイン」見出し・入力・送信・区切り・ゲストログイン）。

  • メール/パスワード: signIn() 成功後、emailVerified が false なら /user/verify-email へ、確認済みなら / へ。Firebase のエラーコードを日本語文言へ マップして表示（例 auth/wrong-password → 「メールアドレスまたはパスワードが正しくありません」）。
  • ゲスト: GuestLoginButton が Firebase 匿名認証（signInAsGuest()）→ /api/User/setGuestRole でロール付与 → クレーム反映待ち（約 1 秒）→ トークン強制リフレッシュ → / へ。ゲストはお気に入り・視聴履歴・プロフィールが無効化される。
  • フォームは isFormValid（両フィールド非空）で送信ボタンの活性を制御し、送信中は loading でスピナー表示。

5. コンポーネント設計（atomic design）

src/components/atomicDesign 配下を atoms / molecules / organisms / pages の 4 層で構成する。 App Router のルート（app/**/page.tsx）が最上位のテンプレートとして各層を束ねる。

5.1 カード 2 種の使い分け

一覧カードは用途で 2 種類ある。横スクロール（ホーム）は ContentCard、グリッド （シリーズ/エピソード内一覧）は GenreContentCard。両者はサムネイル・ランクバッジ・ 三点メタという情報構造を共有しつつ、レイアウトが異なる。

親ContentCardList（flex-row overflow-x-auto）GenreContentCardList（grid ... auto-fill minmax(200px,250px)）
方向常に縦積み（画像上・文字下）モバイル横並び → sm: 以上で縦積み（flex-row sm:flex-col）
最小幅min-w-[164px]min-w-[128px]（グリッドのセルに追従）
用途ホームのランキング行シリーズ内・シーズン別一覧

5.2 モーダルとフィードバック

  • ConfirmationModal: Headless UI の Dialog ベース。タイトル/本文/確認・取消ボタン、 hideButtons でボタン無し表示（ゲスト制限のカウントダウン用）に切替可能。バックドロップ bg-black/30 dark:bg-black/50。
  • トースト: react-hot-toast（Toaster position="bottom-right" duration:4000）で ログイン/ログアウト/エラーを通知。
  • ErrorState: 大きめ絵文字アイコン + タイトル + 説明 + 「再度読み込む」/「ホームに戻る」。 エピソード・シリーズの取得失敗時に全画面表示。
  • LoadingSkeleton: animate-pulse のスケルトン（プロフィール系で使用）。

6. テーマ（next-themes・ライト/ダーク）

テーマは next-themes の ThemeProvider attribute="class"（providers.tsx）で管理し、 Tailwind の dark: バリアントと連動する。切替 UI は ThemeToggleSwitch （useTheme() の resolvedTheme/setTheme）で、ヘッダのサイドパネル「設定」内に置く。 SSR とクライアントの不一致を避けるため mounted フラグが立つまでトグルを描かず、 <html suppressHydrationWarning> を併用する。

局面実装
テーマ属性class（html に dark クラスを付与）
永続化next-themes（providers.tsx の ThemeProvider）が localStorage に保存（既定挙動）
配色Tailwind の dark: ユーティリティ（例 bg-white dark:bg-black / text-gray-900 dark:text-white）
地色body に dark:bg-black dark:text-white、カードは dark:bg-black 中心
初期不一致対策suppressHydrationWarning + ThemeToggleSwitch の mounted ガード
アプリ本体のテーマ（next-themes / Tailwind dark:）と、本ドキュメント HTML 自身のテーマ （data-theme + CSS 変数のトグル）は別系統。前者はアプリの実行時、後者は本書の閲覧時のもの。

ブレークポイントは Tailwind 既定（sm=640px / lg=1024px / xl=1280px）。 モバイルファーストで、要所を sm:/lg:/xl: で拡張する。

8. ゲスト 30 秒制限の UX

匿名（ゲスト）ユーザーはエピソードを 30 秒まで視聴できる。実装は episode/[episodeId]/page.tsx に集約され、 状態は showInitialWarning（入室時の告知）・showLimitModal（制限到達）・ isPlaying・countdown で管理する。

  1. 入室時の告知: ゲストがエピソードを開くと showInitialWarning=true となり、 ConfirmationModal で「ゲストは 30 秒まで。全編はログイン/登録を」を表示。表示中は再生を止める。
  2. 視聴中の監視: VideoPlayer の onProgress（progressInterval=500）が playedSeconds を返し、handleProgress が isAnonymous && playedSeconds >= 30 を検知したら isPlaying=false、showLimitModal=true。
  3. 制限到達: showLimitModal のモーダルは hideButtons でボタンを持たず、 countdown（10 → 0）を毎秒減算。0 で router.push('/') によりホームへ自動遷移。
  4. 再生ガード: いずれかのモーダル表示中は playing={isPlaying && !showLimitModal && !showInitialWarning}、 controls も同条件で無効化し、モーダル越しの操作を防ぐ。
図5 のテキスト説明（スクリーンリーダ向け）

ゲスト（匿名）ユーザーがエピソードに入室すると isAnonymous を判定し、初回警告モーダル （showInitialWarning）を表示する。「了解」で再生を開始し、onProgress（500ms 間隔）で 再生位置を監視する。playedSeconds ≥ 30 に達すると再生を停止して制限モーダルを表示し、 countdown を 10 から 0 へ毎秒減算、0 で router.push('/') によりホームへ自動遷移する。 ログイン / 登録ユーザーはこの制限を通らない。

9. 動画再生（react-player）

再生は atom VideoPlayer が react-player（v2.16）をラップして担う。SSR での不整合を避けるため useEffect で clientSide を立ててから ReactPlayer を描画し、それまでは 「読み込み中...」を出す。親から url・playing・controls・onPlay・ onProgress を受ける。 

<ReactPlayer
  url={url}                       // Streaks 解決の m3u8（HLS）
  controls={controls}            // モーダル中は false
  playing={playing}              // ゲスト制限・モーダルで制御
  width='100%' height='100%'
  onPlay={handlePlay}            // 視聴履歴の記録トリガ（非ゲスト）
  onProgress={onProgress}        // playedSeconds を親へ
  progressInterval={500}
  config={{ file:{ attributes:{ controlsList:'nodownload', playsInline:true } } }}
/>
  • HLS ソース: url は StreamService 経由で解決された video_url（m3u8）。 react-player は HLS を file プレイヤーで扱う。
  • 視聴履歴: onPlay → handleVideoPlay が非ゲスト時に recordHistory() で エピソード情報を記録し、共有 Context（WatchHistoryDataContext）にも追加してヘッダの履歴一覧へ即反映。
  • 属性: controlsList:'nodownload' でダウンロード UI を抑制、playsInline:true で モバイルのインライン再生を有効化。
ストリーム解決の位置づけ（データ層の事実）: video_url は BFF 側で TVER の公開 API フロー （session → ranking → callEpisode → statics JSON → streaks_info → playback）を辿って得た HLS。 現行の主経路は Streaks（2025 年時点）。Streaks の Playback 呼び出しは Origin: https://tver.jp が必須で、 これを間違えると 403 id139 が返る。API キーは streaks_info_v2.json の月替わりキー（idx = (JST の月 % 6) || 6 → key0{idx}）を用いる。 Streaks を持たない旧作（稀少）は Brightcove（accountID / videoRefID + policyKey）が経路になり、段階 2 以降の対応予定。 UI 層はこの結果 URL を受け取って再生するだけで、解決ロジックは持たない。

11. 改善提案（客観）

以下はコードから観測できる事実に基づく提案で、優先度は工数と影響の目安。是非の評価ではない。

11.1 提案一覧

#提案根拠（観測事実）目安
1 ホームの認証ガードをクライアント router.push からルート境界（middleware / サーバ判定）へ寄せる Main.tsx は useEffect で未認証を検知して遷移するため、初回に一瞬ホームが描かれてから飛ぶ経路がある 中
2 字幕トラック（tracks[] captions/subtitles）を VideoPlayer へ渡し、字幕トグルを追加 ストリーム解決レイヤーで字幕トラック（ResolvedStream.subtitles）を抽出済みだが、VideoPlayer の config は attributes のみで tracks の受け取り未対応 中
3 プレイヤーの寸法に 100vw ではなく 100% ＋アスペクト比指定を検討 width:100vw は縦スクロールバー幅を含むため、横方向にわずかなはみ出し（横スクロール）が生じ得る 小
4 ランクバッジ画像の外部依存（tver.jp の SVG 直リンク）をローカル資産化またはフォールバック付与 ContentCard/GenreContentCard がレンダ時に外部 URL を都度読み込む 小
5 横スクロール一覧にスクロール操作（ボタン/キーボード）と aria ロールを付与 ContentCardList は overflow-x-auto のみで、ポインタ以外の操作導線が無い 中
6 素の「Loading...」を既存 LoadingSkeleton（またはカード形状スケルトン）へ置換 ホーム/エピソードは文字のみ。スケルトンは実装済みだが一覧未使用でレイアウトシフトが起きやすい 小
7 カウントダウン・トグル等の動的テキストに aria-live、ロゴ sr-only をアプリ名へ修正 制限モーダルの残秒表示が読み上げに乗らない。ロゴ代替が「Your Company」のまま 小
8 未使用/プレースホルダ UI（TabsWithUnderline の Lorem ipsum 等）の整理または実装完了 ダミー本文と不完全な ARIA を持つコンポーネントが残存 小
9 カード副次テキストの階調をダーク時に一段明るく（gray-400/500 → gray-300 等）調整 小さめ副次テキストの階調差が小さい箇所がある 小
