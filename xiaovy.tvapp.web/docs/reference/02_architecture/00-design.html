対象 tvapp-web (xiaovy.tvapp.web) ブランチ project/tvapp-cf-zero-cost 関連 Platform-Stream-Loader(除去対象) / TVER 公開API / Firebase / Streaks スタック Next.js 14.2 App Router / TS strict / Tailwind ステータス WIP(移行設計)
本書は tvapp-web の詳細設計を扱う。レイヤ構成、DI(依存注入)、ストリーム解決モジュール(純TS)、 認証、Cloudflare/OpenNext 構成、firebase-admin/Cosmos 除去方針を、実コードとブリーフの事実に基づいて記述する。 記述は客観的事実(技術・構成・依存・データフロー)に徹し、題材の是非は扱わない。

1. 概要とスコープ

tvapp-web は Next.js 14.2 の App Router 構成で、フロントエンド(React クライアント)と BFF(App Router の API ルート)を 1 リポジトリに同居させる。TVER の公開 API を BFF が代理し、フロントは自ドメインの /api/* だけを叩く。 現行(version 1.4.0)は次の 3 系統に依存している。

  • TVER 公開 API 代理: セッション発行・ランキング・エピソード情報・静的コンテンツ・検索。
  • ストリーム解決: 現状は外部 Azure Functions(Platform-Stream-Loader / yt-dlp)へ委譲。
  • ユーザーデータ / 認証: Firebase Auth(クライアント)+ firebase-admin(サーバ: Firestore / Storage / ID トークン検証)、 課金メタは Cosmos DB。

本設計の目的は、この構成を Cloudflare Workers + @opennextjs/cloudflare 上で動く形へ移行し、 ストリーム解決を純 TypeScript で API ルート内に内製化して Azure Functions / yt-dlp / Cosmos を除去することにある。 MVP は「TVER 閲覧 + 純TS再生 + Firebase Auth(login/register/guest)」で、Firestore を必要とせず即時に成立する。 お気に入り / 視聴履歴 / プロフィール / 管理画面は段階 2 とする。

2. レイヤ構成

クライアントは components(Atomic Design)→ hooks → contexts(DI 境界)→ services(I*Service + implementation) の順に依存し、services は自ドメインの app/api ルート(BFF)を fetch する。BFF から先が TVER / Firebase / Streaks などの外部境界である。関心事を層で分離し、UI から外部 API の詳細を隠蔽するのが基本方針。

3. サービス層とインタフェース分離

各ドメインは src/services/I<Name>Service.ts にインタフェースを、 src/services/implementation/<Name>Service.ts に実装クラスを置く (ファイル名の実例: ISessionService.ts / SessionService.ts。シリーズのみ実装ファイルが SeriseService.ts という綴りだが、公開クラス名は SeriesService)。実装は自ドメインの API ルートを叩く薄いクライアントで、 TVER などの外部詳細は BFF 側に閉じる。

例として IStreamService は getVideoUrl(episodeId) の 1 メソッドだけを公開する。

// src/services/IStreamService.ts
export interface IStreamService {
  getVideoUrl: (episodeId: string) => Promise<StreamResponseType>;
}

// src/services/implementation/StreamService.ts(抜粋)
export class StreamService implements IStreamService {
  async getVideoUrl(episodeId: string): Promise<StreamResponseType> {
    const url = `/api/service/call/streaminglink?` +
      new URLSearchParams({ episodeId }).toString();
    const response = await fetch(url, {
      method: 'GET',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${localStorage.getItem('IdToken')}`,
      },
    });
    if (!response.ok) throw new Error('インターネット接続がありません');
    return (await response.json()) as StreamResponseType;
  }
}

同様に SessionService.getSessionToken() は POST /api/service/session を叩いて { platformUid, platformToken } を得る。サービス実装は「1 メソッド = 1 API ルート」対応を基本とし、 外部 API の合成やヘッダ付与(Origin など)は必ず BFF 側で行う。

4. DI:providers.tsx によるコンテキスト注入

依存注入は React Context で行う。src/contexts/<Name>Context.ts が createContext<I<Name>Service | null>(null) を定義し、 src/app/providers.tsx が実装インスタンスを useMemo で生成して各 Provider の value に流し込む。 hooks は useContext(...Context) でインタフェース越しに実装を受け取り、具象クラスに直接依存しない。 

// src/app/providers.tsx(抜粋)
const streamService = useMemo(() => new StreamService(), []);   // 再生成防止

return (
  <ThemeProvider attribute="class">
    <AuthProvider>
      <FavoritesDataProvider>
        <WatchHistoryDataProvider>
          <SessionServiceContext.Provider value={sessionService}>
            {/* … TvHome / Series / Episode / Ranking … */}
            <StreamServiceContext.Provider value={streamService}>
              {/* … Profile / Favorite / WatchHistory … */}
              <Toaster position="bottom-right" />
              {children}
            </StreamServiceContext.Provider>
          </SessionServiceContext.Provider>
        </WatchHistoryDataProvider>
      </FavoritesDataProvider>
    </AuthProvider>
  </ThemeProvider>
);

hooks 側は次のようにインタフェース越しに使う(useStreamService)。null チェックを経てから getVideoUrl を呼ぶため、Provider 未設定時も型安全に扱える。 

// src/hooks/useStream.ts(抜粋)
const streamService = useContext(StreamServiceContext);  // IStreamService | null
useEffect(() => {
  if (!episodeId || !streamService) return;
  streamService.getVideoUrl(episodeId).then(setStreamUrl).catch(console.error);
}, [episodeId, streamService]);

5. ストリーム解決モジュール設計(純TS)

移行の中心は、Azure Functions + yt-dlp に委譲していたマニフェスト解決を、フレームワーク非依存の純 TypeScript として BFF 内に内製化することである。TVER の公開 API を実測フロー通りに順に叩き、DRM 無しの HLS(m3u8)ソースを得る。 fetch と Web 標準 API のみを使い、Node 固有 API に依存しない(Workers 互換)。

5.1 解決フロー(6 ステップ)

各ステップの完全な URL と主なヘッダを示す。Origin ヘッダが必須(欠落・不一致で 403 id139)なのは Step 6(Playback)のみで、セッション作成(Step 1)の Origin/Referer は https://s.tver.jp、それ以外の TVER API は https://tver.jp を用いる。

ステップメソッド / 完全 URL主なヘッダ
1. セッション作成POST https://platform-api.tver.jp/v2/api/platform_users/browser/create(body device_type=pc)Origin/Referer: https://s.tver.jp
2. エピソード情報GET https://platform-api.tver.jp/service/api/v1/callEpisode/{id}?platform_uid=&platform_token=Origin/Referer: https://tver.jp
3. 静的コンテンツGET https://statics.tver.jp/content/episode/{id}.json?v={version}—

5.3 キー選択

キー選択は「JST の月」を Intl.DateTimeFormat で決定論的に求め、idx = (月 % 6) || 6 で key01..key06 を選ぶ。

// stream/keySelector.ts
export function selectStreaksKey(keys: StreaksKeys, now = new Date()): string {
  const jstMonth = Number(new Intl.DateTimeFormat('en-US', {
    timeZone: 'Asia/Tokyo', month: 'numeric',
  }).format(now));                       // 1..12(JST)
  const idx = (jstMonth % 6) || 6;       // 1..6
  return keys[`key0${idx}` as keyof StreaksKeys];
}
実装(streamResolver.ts)では Intl.DateTimeFormat ではなく、UTC の月に JST オフセット(UTC+9)を加味した Date.getUTCMonth() ベースで同じ月を求めている。両アプローチは JST の月を得る点で等価であり、選択されるキーは一致する。上のコードはより明示的なタイムゾーン指定として並記する。

5.4 Playback 呼び出しとソース選別

Playback は Origin: https://tver.jp が必須(欠落・不一致で 403 id139)。返却 sources[] のうち type が m3u8 系 MIME(application/x-mpegurl / application/vnd.apple.mpegurl。大文字小文字は無視)かつ key_systems を持たない(非DRM)ものを採用する。字幕は tracks[](kind が captions / subtitles)から抽出する。 

// stream/playback.ts(抜粋)
const res = await fetch(
  `https://playback.api.streaks.jp/v1/projects/${projectID}/medias/ref:${videoRefID}`,
  { headers: {
      'Accept': 'application/json',
      'Origin': 'https://tver.jp',   // 必須(欠落で 403 id139)
      'Referer': 'https://tver.jp/',
      'X-Streaks-Api-Key': apiKey,
  }},
);

// stream/resolve.ts(抜粋 / オーケストレータ)
export async function resolveStream(episodeId: string): Promise<ResolvedStream> {
  const { platformUid, platformToken } = await createSession();
  const version = await callEpisode(episodeId, platformUid, platformToken);
  const statics = await fetchStatics(episodeId, version);       // streaks{projectID, mediaID, videoRefID}
  const keys    = await fetchStreaksInfo();                     // キャッシュ済みなら即時
  const apiKey  = selectStreaksKey(keys[statics.streaks.projectID]);
  const pb      = await resolvePlayback(statics.streaks, apiKey);
  const M3U8_TYPES = new Set([                                  // m3u8 系 MIME(実装の M3U8_TYPES 相当)
    'application/x-mpegurl', 'application/vnd.apple.mpegurl',
  ]);
  const source  = pb.sources.find(
    (s) => M3U8_TYPES.has(s.type?.toLowerCase()) && !s.key_systems, // m3u8 かつ非DRM
  );
  if (!source) throw new StreamError('MEDIA_NOT_FOUND');
  return { m3u8: source.src, captions: pickCaptions(pb.tracks) };
}

解決過程のエラーは、外部コードを内部分類へマッピングして UI に返す。

5.6 キャッシュ方針

  • streaks_info(月替わりキー): 変動が緩やかなため Workers KV もしくはメモリ + 短期 TTL でキャッシュ。 キー選択はキャッシュ済みキー集合に対しリクエスト時の JST 月で毎回算出する。
  • statics JSON: episodeId + version で内容が確定するため、その組で長めにキャッシュ可能。
  • セッショントークン: リクエスト単位で取得、または短期キャッシュ。
  • id126(キー不正)検出時は streaks_info キャッシュを破棄して再取得する。

5.7 フォールバック経路(Brightcove)

静的コンテンツ JSON(Step 3)は streaks に加え video(accountID / videoRefID / playerID = Brightcove)も返す。 streaks を持たない旧作は Brightcove(account / videoRefID + policyKey)を経路とする。現行コンテンツの主経路は Streaks であり、本設計は Streaks 経路を対象に純TS解決を設計する。

6. 認証設計:Firebase Auth + jose 検証

クライアント認証は Firebase Auth(client SDK)を維持する。src/lib/firebase.ts は ブラウザかつ apiKey 存在時のみ初期化し、AuthContext が signInWithEmailAndPassword / createUserWithEmailAndPassword / signInAnonymously(ゲスト)/ onAuthStateChanged を提供する。ログイン成功時に ID トークンを localStorage (NEXT_PUBLIC_IDTOKEN_NAME)へ保存する暫定実装がある。

用語の整理(重要): 現行 utils/verify-token は jose.jwtVerify を JWT_SECRET_KEY(HMAC 対称鍵)で使う自前 JWT の検証であり、Firebase ID トークンとは別物。 移行で置換対象になるのは、firebase-admin の verifyIdToken()(RS256 / Google 公開鍵)である。

6.1 サーバ側 ID トークン検証の移行

firebase-admin は Node 前提(gRPC など)で Workers 非互換のため、サーバでの Firebase ID トークン検証は jose + Google 公開鍵(JWKS)へ移す。jose は既存依存で、fetch + Web Crypto のみを使い Workers 互換。 

import { jwtVerify, createRemoteJWKSet } from 'jose';

// Firebase ID トークンの署名鍵(JWK / RS256)
const JWKS = createRemoteJWKSet(new URL(
  'https://www.googleapis.com/service_accounts/v1/jwk/securetoken@system.gserviceaccount.com'
));

export async function verifyFirebaseIdToken(idToken: string, projectId: string) {
  const { payload } = await jwtVerify(idToken, JWKS, {
    issuer: `https://securetoken.google.com/${projectId}`,
    audience: projectId,
  });
  return payload;   // sub(uid), email, firebase.sign_in_provider など
}

これにより、認可が必要な API ルート(User/* や管理系)は firebase-admin を介さずトークンを検証できる。 ゲスト(匿名)のロール付与はトークンのクレームで判別する。

7. Cloudflare / OpenNext 構成

ホスティングは Cloudflare Workers + @opennextjs/cloudflare。OpenNext が Next.js の App Router 出力を Workers ランタイム向けにアダプトする。nodejs_compat フラグを有効化し、Web 標準 API(fetch / Web Crypto)を用いる。 無料枠は 10 万リクエスト/日。

項目内容
ランタイムCloudflare Workers(V8 isolate。Node ではない)
アダプタ@opennextjs/cloudflare(OpenNext)
互換フラグnodejs_compat
デプロイwrangler(build → deploy)
キャッシュWorkers KV(streaks_info / statics 等の短期キャッシュに利用可)
移行の設計上の含意: BFF の各ルートは Node 固有 API を使わないこと。ストリーム解決(§5)と ID トークン検証(§6)は いずれも fetch + Web Crypto のみで完結するよう設計しており、この制約を満たす。

8. firebase-admin / Cosmos 除去方針

src/lib/firebase-admin.ts は adminAuth / adminDb(Firestore)/ adminStorage を初期化するが、 gRPC ベースの Firestore admin は Workers 非互換。段階的に次の方針で除去する。

対象現行移行後段階
ID トークン検証firebase-admin verifyIdTokenjose + Google JWKS(§6.1)MVP
ユーザーデータ(お気に入り/履歴/プロフィール)firebase-admin Firestore(gRPC)Firestore client SDK + セキュリティルール(本人所有データ直読み書き)段階2
画像firebase-admin StorageFirebase Storage client / ルール段階2
課金メタCosmos DB(/api/cosmosdb/videos)MVP 除外 → 段階的に Firestore へ後続

MVP では Firestore を必要とせず、TVER 閲覧・純TS再生・Firebase Auth(login/register/guest)のみで成立する。 ユーザーデータ層は段階 2 でクライアント SDK + セキュリティルールへ移し、サーバ(Workers)からは gRPC 依存を持たない形にする。

9. 移行チェックリスト

  • service/call/streaminglink を Azure fetch から resolveStream() 呼び出しへ差替。
  • 純TS モジュール(session / callEpisode / statics / streaksInfo / keySelector / playback / resolve / errors)を追加し、Node 固有 API を排除。
  • content/episode/[episodeId] に ?v={version} を付与。
  • Playback で Origin: https://tver.jp を固定付与、key_systems 付きソースを除外。
  • 月替わりキー選択 idx = (JST月 % 6) || 6 を実装、id126 時にキャッシュ破棄。
  • サーバ ID トークン検証を firebase-admin から jose + Google JWKS へ置換。
  • firebase-admin / Cosmos 依存ルートを MVP から切り離し(段階 2 で Firestore client + ルール)。
  • @opennextjs/cloudflare + nodejs_compat でビルド・デプロイ、KV キャッシュ配線。
