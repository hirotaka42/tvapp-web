1. 概要と全体像

TVER をより手軽に見るための Web アプリ。3 つのマネージドサービスを役割分担させ、 全体をほぼ $0 で運用する。

役割基盤要点
フロント + ブラウズ
（ランキング / 検索 / シリーズ）		 Cloudflare Workers（OpenNext） Next.js 16（App Router）を @opennextjs/cloudflare で Workers に載せる。 無料枠 10 万リクエスト/日。TVER 公開 API の代理（ブラウズ）は Workers から直接成立する。
認証 Firebase Authentication Google / メール+パスワード / メールリンク（passwordless）。匿名（ゲスト）は不採用。 Spark（無料）プランのまま運用。
再生（ストリーム解決） Azure Function
（Platform-Stream-Loader）		 Japan East / Consumption（Y1）従量課金・無料枠内（~$0）。 Cloudflare Workers の外向き IP は非 JP のため Streaks の地域制限（id124=geo）で弾かれる。 JP-egress の Azure Function へ委譲することで本番の再生解決が成立する。

ストリーム解決の二段構え

/api/service/call/streaminglink（src/app/api/service/call/streaminglink/route.ts）は 環境変数の有無で解決経路を切り替える。

  1. AZURE_FUNCTION_STREEAMING が設定されていれば、Azure Function （/api/backend_stream_url_http?code=…&url=…&service_id=1&res_type=6）へ委譲する（本番）。
  2. 未設定なら、純 TypeScript の TVER→Streaks 解決 （src/lib/tver/streamResolver.ts）を使う（ローカル / JP-egress ホスト向け）。

2. 開発ワークフロー（順序が重要）

開発は次の順序で進める。ローカル開発がデフォルトで、本番反映は最後。

  1. ローカルで開発する（デフォルト）— npm run dev。TVER 公開 API には ローカルから直接つながる（JP 家庭回線なら純 TS 解決で再生まで確認できる）。
  2. 本物の API に繋いで実機確認する — TVER 公開 API はそのまま実物。Azure 経由の解決を確認したいときは .env.local に AZURE_FUNCTION_STREEAMING（+ key）を設定する。 bash scripts/health-check.sh で実データ経路（session → streaminglink → m3u8）まで検証する。
  3. 問題なければ main へマージする — PR / push で CI（型検査 + テスト + カバレッジ 80% + ビルド）が走り、 通らない変更は main に入れない（ブランチ保護）。
  4. Cloudflare 本番で最終確認する — main へのマージで package.json の version が 上がっていれば版ゲートが自動デプロイする。デプロイ後は本番 URL の表示・ログイン・再生と /api/health の build_sha を確認する。
版ゲート方式のため、version を上げないマージはデプロイされない（AGENTS-SHARED §4）。 出荷物が変わる変更では xiaovy.tvapp.web/package.json の version を上げてからマージする。

3. ローカル実行手順

3.1 前提

  • Node.js 20+ と npm（CI も Node 20 を使用）。
  • 作業ディレクトリはアプリ本体 tvapp-service/tvapp-web/xiaovy.tvapp.web。

3.2 インストールと環境変数

cd xiaovy.tvapp.web
npm install
cp .env.local.example .env.local   # 値を記入（下表）
変数必須内容
NEXT_PUBLIC_FIREBASE_API_KEY
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN
NEXT_PUBLIC_FIREBASE_PROJECT_ID
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID
NEXT_PUBLIC_FIREBASE_APP_ID		 必須（6 値） Firebase コンソール > プロジェクトの設定 > 全般 > SDK 構成。 クライアントに埋め込まれる公開値（秘密情報ではない）。
NEXT_PUBLIC_IDTOKEN_NAME必須 認証トークンの localStorage キー名（例 IdToken）。
NEXT_PUBLIC_DEV_BYPASS_AUTH任意（開発専用） 1 で認証を通さずに UI を確認できる（モックユーザーでログイン済み扱い）。 コード側が NODE_ENV !== 'production' との二重ガードで、本番では絶対に無効。
AZURE_FUNCTION_STREEAMING
AZURE_FUNCTION_STREEAMING_CODE_KEY		 任意 設定するとストリーム解決を Azure Function へ委譲（本番と同じ経路）。 未設定ならローカルの純 TS 解決を使う。
NEXT_PUBLIC_STREAM_PASSWORD ほか任意 ライブ配信（/stream）・独自 JWT を使う場合のみ（.env.local.example 参照）。

3.3 起動

npm run dev
# → http://localhost:3000
  • TVER のブラウズ（ランキング / 検索 / シリーズ）は本物の TVER 公開 API に接続する。
  • 再生解決は AZURE_FUNCTION_STREEAMING 設定時は Azure 経由、 未設定時はローカルの純 TS 解決（JP 回線から動作）。
  • Firebase の承認済みドメインは localhost が既定で許可されているため、 ローカルの Google ログインは追加設定なしで動く。
dev サーバーは使い終わったら必ず停止する。起動時に PID を控え、検証後に kill し、 ps と ポート解放で残存ゼロを確認する（orphan 化した next dev が CPU を飽和させた事例あり。CLAUDE.md 教訓参照）。

3.4 テストと動作確認

コマンド内容
npm testユニットテスト一括実行（vitest run）。ライブ疎通テスト（*.live.test.ts）は既定で除外。
npm run test:watchwatch モード。
npm run test:coverageカバレッジ付き実行。src/lib/** / src/utils/** に対して lines / functions / branches / statements 各 80% 未満で失敗（CI・デプロイと同じゲート）。
npm run test:liveTVER_LIVE=1 で実 TVER API への疎通テストも実行（ネットワーク必須・JP 回線推奨）。
bash scripts/health-check.sh [PORT]ローカル生存チェック（既定ポート 3000）。 プロセス/ポート、/api/health・/・/user/login の HTTP 200、 TVER 実データ経路（session 発行 → ランキングから実エピソード → streaminglink が m3u8 を返す）、 ログのエラー走査（LOG=path 指定時）を検証。終了コード 0=健全 / 1=異常。

3.5 Cloudflare 向けビルドの事前確認（任意）

npm run cf:build     # OpenNext で Workers バンドル生成（.open-next/）
npm run cf:preview   # ビルド + ローカルプレビュー（workerd 実行）

4.3 手動デプロイは deploy:safe（必ず再ビルド + 鮮度照合）

npm run deploy:safe   # = bash scripts/deploy.sh

scripts/deploy.sh の保証（「デプロイしたのに中身が古い」事故の構造的防止）:

  1. 未コミット変更（src / package.json / wrangler.jsonc 等）があれば中断。
  2. 古い .open-next / .next を必ず削除して新規ビルド（npm run cf:build）。
  3. wrangler deploy --var BUILD_SHA:<HEAD短SHA> で SHA を埋め込みデプロイ。
  4. デプロイ後、本番 /api/health の build_sha が HEAD と一致するか照合 （エッジ反映待ちで最大 ~48 秒リトライ）。不一致なら失敗させる。
opennextjs-cloudflare deploy 単体や bare wrangler deploy は 再ビルドせず既存 .open-next を上げるだけ。手動デプロイは必ず deploy:safe を使う （cf:deploy も build && deploy に修正済みだが、鮮度照合まで行うのは deploy:safe のみ）。

5. コスト（$0 方針）

  • Cloudflare: Workers 無料枠（10 万リクエスト/日）。有料アドオンを付けない。
  • Firebase: Spark（無料）プランのまま。Blaze（従量課金）へ切り替えない。 使うのは Authentication（無料）と、段階2で Firestore 無料枠のみ。 Storage は使わない（新規プロジェクトで Blaze 必須になり得るため）。Cloud Functions も使わない。
  • Azure: Consumption（Y1）の無料枠内（~$0）。既存 Function の再利用で新規費用なし。
  • 課金につながる操作（Blaze 化・カード登録・従量サービス有効化）は行わない。

6. トラブルシュート

症状原因と対処
本番で Google ログインができない
（ポップアップ即閉じ / auth/unauthorized-domain）		 Firebase の承認済みドメインに workers.dev の本番ドメインが未追加。 Authentication > Settings > 承認済みドメインに追加する（§4.4）。localhost は既定許可のためローカルでは再現しない。
再生解決が「この地域からは視聴できません」
（id124 = geo）		 Streaks が非 JP / データセンター IP を拒否する。Cloudflare Workers 直の解決は成立しないため、 本番は AZURE_FUNCTION_STREEAMING（JP-egress の Azure）委譲が前提。 ローカルでも国外 IP・VPN 経由では純 TS 解決が同様に弾かれる（JP 回線で確認する）。
デプロイしたのに本番の中身が古い 再ビルドなしのデプロイ（bare wrangler deploy 等）で古い .open-next を上げた。 手動デプロイは必ず npm run deploy:safe（削除 → 再ビルド → BUILD_SHA 照合。§4.3）。 本番 /api/health の build_sha と HEAD の一致で確認できる。
main にマージしたのにデプロイされない 版ゲート仕様（version 変化なし = スキップ）。package.json の version を上げてマージする（§2）。
CI がカバレッジで落ちる src/lib/** / src/utils/** の 4 指標いずれかが 80% 未満。 ローカルで npm run test:coverage を実行して不足箇所を特定する。
ローカルで Firebase 未設定のまま UI を確認したい .env.local に NEXT_PUBLIC_DEV_BYPASS_AUTH="1"（開発専用・本番では二重ガードで無効。§3.2）。
