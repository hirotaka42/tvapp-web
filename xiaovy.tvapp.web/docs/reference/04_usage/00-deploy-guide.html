0. これは何をデプロイするか

  • フロント/BFF: Next.js 14（App Router）を @opennextjs/cloudflare（OpenNext）で Cloudflare Workers に載せる。
  • データ解決: TVER の閲覧・再生は公開 API への純 TypeScript 解決で行う。 Azure Functions / yt-dlp / Cosmos DB は不要。
  • 認証: Firebase Auth（クライアント）。
  • MVP スコープ: ホーム（ランキング閲覧）／検索／シリーズ／エピソード再生／ログイン・新規登録・ゲスト。
  • 段階2（お気に入り／視聴履歴／プロフィール／管理）は現状 deferred-phase2/ に退避済みで MVP には含めない（Firestore クライアント方式で後日有効化予定）。
  • コスト: Cloudflare Workers 無料枠（10 万リクエスト/日）、Firebase Spark（無料枠）。ほぼ $0。

1. 事前に用意するもの

  • Firebase プロジェクト（用意済み）。Authentication を有効化し、サインイン方法で 「メール/パスワード」と「匿名」を ON にする。
  • Firebase の SDK 構成値6つ（コンソール > プロジェクトの設定 > 全般 > マイアプリ > SDK setup and configuration）: apiKey / authDomain / projectId / storageBucket / messagingSenderId / appId。
  • Cloudflare アカウント（無料）。デプロイには wrangler のログイン（npx wrangler login） または API トークン（環境変数 CLOUDFLARE_API_TOKEN）。
  • Node.js 20+ と npm。

2. 環境変数（重要な性質）

.env.local.example を .env.local にコピーして Firebase 値を記入する。

cp .env.local.example .env.local
# エディタで .env.local を開き、Firebase の6値 + IDTOKEN_NAME を記入

MVP で必須の変数

すべて NEXT_PUBLIC_ 接頭辞で、ビルド時にクライアントへ埋め込まれる。

変数名内容
NEXT_PUBLIC_FIREBASE_API_KEYSDK 構成の apiKey
NEXT_PUBLIC_FIREBASE_AUTH_DOMAINauthDomain（例 your-project.firebaseapp.com）
NEXT_PUBLIC_FIREBASE_PROJECT_IDprojectId
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKETstorageBucket（例 your-project.appspot.com）
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_IDmessagingSenderId
NEXT_PUBLIC_FIREBASE_APP_IDappId
NEXT_PUBLIC_IDTOKEN_NAME認証トークンの localStorage キー名（任意の文字列。例 IdToken）
つまずきやすい点: NEXT_PUBLIC_* は「ビルド時」に埋め込まれる。 そのため npm run cf:build を実行する環境（手元 or CI）に必ず設定しておくこと。 実行時の Worker 変数（wrangler の vars 等）に入れても反映されない。

サーバー専用の秘密（将来 JWT_SECRET_KEY 等）は npx wrangler secret put NAME か wrangler.jsonc の vars/secrets で扱う。 MVP では不要。

3. ローカル検証手順

作業ディレクトリ: tvapp-service/tvapp-web/xiaovy.tvapp.web。

  1. 依存をインストール: 
    npm install
  2. ユニットテスト（vitest 34 件・ネット不要）: 
    npm test
  3. （任意）実 TVER への疎通テスト（日本国内 IP 必須）: 
    npm run test:live
  4. 開発サーバーで動作確認 → http://localhost:3000 でログイン → ホーム → エピソード再生: 
    npm run dev
  5. Cloudflare 向けビルド（next build → OpenNext で .open-next/worker.js を生成）: 
    npm run cf:build
  6. ローカルの workerd で Worker を起動して確認（wrangler dev 相当）: 
    npm run cf:preview

4. Firebase コンソール設定

  • Authentication > Sign-in method:「メール/パスワード」と「匿名」を有効化する。
  • Authentication > Settings > 承認済みドメイン: デプロイ先ドメインを追加する。
    • <worker名>.<サブドメイン>.workers.dev
    • 独自ドメインがあればそれも追加
    • localhost は既定で許可されている（ローカル検証はそのまま可）

5. Cloudflare へデプロイ

  1. 初回のみ: wrangler をログイン（ブラウザで承認）。CI では CLOUDFLARE_API_TOKEN を使う: 
    npx wrangler login
  2. デプロイ（内部で opennextjs-cloudflare build → deploy）。 .env.local の NEXT_PUBLIC_* がビルドに乗る: 
    npm run cf:deploy
  3. 完了後、払い出される URL を開く: 
    https://<worker名>.<サブドメイン>.workers.dev

wrangler.jsonc の設定値（現行）:

キー値
nametvapp-web
main.open-next/worker.js
compatibility_flags["nodejs_compat"]

6. デプロイ後の確認（完了定義）

  1. 払い出し URL でログイン（メール/パスワード or ゲスト）ができる。
  2. ホームにランキングが表示される。エピソードを開くと再生できる。
  3. ゲストは 30 秒で視聴制限（現行仕様）。
つまずきやすい点: Firebase の承認済みドメインに workers.dev を追加していないと、 払い出し URL でのログインが失敗する。ログイン画面から先に進めないときは、まずここを確認する（§4）。

8. コスト・制限・トラブルシュート

コストと制限

  • Cloudflare Workers 無料枠（10 万リクエスト/日）、Firebase Spark（無料枠）で運用でき、ほぼ $0。
  • TVER は日本国内（JP）からのみ視聴可能（geo 制限）。国外からのアクセスは 451 応答になる。

よくある詰まり

ログイン画面から進めないFirebase の承認済みドメイン未追加workers.dev を承認済みドメインへ追加（§4）
