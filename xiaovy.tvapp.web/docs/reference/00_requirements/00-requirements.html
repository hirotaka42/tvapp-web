1. 目的と背景

TVapp は、公開されている TVER の視聴用 Web/HTTP API を素の HTTP で呼び出し、 ランキング・検索・シリーズ・エピソード情報を取得して、ブラウザ上で HLS ストリームを再生する Web アプリケーションである。 現行は Next.js 14.2(App Router)のフロント + BFF(API ルート)を中核に、以下 3 リポジトリで構成される。

リポジトリ役割スタック本移行での位置づけ
tvapp-webフロント + BFF(TVER 代理・ユーザーデータ・再生)Next.js 14.2 / TS strict / Tailwind / Firebase移行の作業ルート(正本)
Platform-Stream-Loaderページ URL → m3u8 マニフェスト解決Azure Functions(Python) + yt-dlp純 TS 実装へ置換して廃止対象
tvapp-bff旧 BFF(2024 停止).NET7移行対象外・廃止候補

本要求整理の主眼は、現行機能を保ちつつ運用構成を Cloudflare + Firebase へ移し、 外部の重い依存(Azure Functions / yt-dlp / Cosmos DB)を除去して ほぼ $0 で運用可能にすることである。 「何を作るか(機能)」「どの品質で成立させるか(非機能)」「どこまでやるか(スコープ)」を確定し、 後続の設計・実装フェーズの入力とする。

本アプリの利用主体・関係者を、Firebase Auth のロール(src/types/User.ts の UserRole)と対応づけて整理する。

ステークホルダロール値関心事 / 権限
視聴者(一般ユーザー)GENERAL(0) 以上ログインして全編視聴・お気に入り・視聴履歴・プロフィール編集。全機能の中心的利用者。
ゲスト(匿名ユーザー)GUEST(-1)匿名サインインで閲覧・再生を試せる。再生は 30 秒制限。お気に入り/履歴/課金系の一部は不可。

運用者(開発/デプロイ担当)—低コスト・低運用負荷での可用性維持。デプロイ・キー管理・監視。
外部提供元(TVER / Streaks / Brightcove / Firebase / Cloudflare)—依存する外部 API・基盤。可用性・仕様は外部要因で変わりうる。

3. 解決したい課題

現行構成を事実として並べ、本移行で解消を狙う課題を課題 ID(P-n)として定義する。

ID現状(事実)狙い
P-1ストリーム解決を外部 Azure Functions + yt-dlp に依存(調査ブリーフ「唯一の重い/課金源」)。解決フローを純 TS で API ルート内に内製し、外部関数を不要にする。
P-2サーバ側ユーザーデータが firebase-admin(gRPC Firestore)前提で、エッジランタイム非互換。ID トークン検証を jose+Google 公開鍵に、データを Firestore client SDK + セキュリティルールへ寄せる。

P-4ホスティング/運用構成が固定費・運用負荷を伴う。Cloudflare Workers + OpenNext(無料枠 10 万 req/日)へ移し、ほぼ $0 に。
P-5旧 BFF(tvapp-bff / .NET7)が 2024 停止のまま残存。移行対象から外し、廃止候補として整理する。

4.2 再生ユースケースの内部データフロー(UC-06 の要点)

再生は TVER の公開 API を段階的に呼び、最終的に Streaks の playback API から非 DRM の HLS(m3u8)を得る。 現行 tvapp-web はストリーム解決部を外部 Azure Functions(/api/service/call/streaminglink)に委譲しているが、 移行後はこの全段を純 TS で内製する。呼び出しヘッダ(特に Origin)と月替わりキー選択が要点である。

4.3 各ステップの詳細(エンドポイント / 主なヘッダ / 主要レスポンス)

6. 非機能要求(NFR)

ID区分要求(測定可能な形)
NFR-01コスト定常運用費を ほぼ $0 に保つ。Cloudflare Workers 無料枠(10 万 req/日)内で成立させ、Azure Functions / yt-dlp / Cosmos DB を廃止する。
NFR-02ホスティング/互換Cloudflare Workers + @opennextjs/cloudflare で動作。nodejs_compat 有効。Node 非互換の依存(gRPC firebase-admin 等)を持ち込まない。
NFR-03可用性再生の主経路は Streaks。旧作は Brightcove へフォールバックし、単一経路の失敗で全滅させない。/api/health で稼働確認できる。
NFR-04地域TVER は日本(JP)限定配信。日本 IP からの疎通を前提とする。月替わりキー選択は JST 基準((JSTの月 % 6) || 6)。
NFR-05互換ヘッダ各外部呼び出しで既定ヘッダを厳守する(Origin はスキーム付き。session=Origin: https://s.tver.jp / ranking・search・series=Origin: https://tver.jp + x-tver-platform-type: web / playback=Origin: https://tver.jp 必須)。
NFR-06認証セキュリティ保護 API は Authorization: Bearer <idToken> を検証する。匿名ユーザーは課金系(/api/cosmosdb/videos)を 403 で拒否する。本人以外のユーザーデータへアクセスさせない。
NFR-07パフォーマンス月替わりキー・公開鍵などの変化の遅い値はキャッシュ(Workers KV or 短期メモリ)して外部呼び出しを削減する。
NFR-08保守性 / テストTypeScript strict を維持し tsc --noEmit が通ること。テスト FW(vitest)を導入し、ストリーム解決の各段(①セッション〜⑥Playback)を単体テストする。期待値は yt-dlp 2026.03.17 の tver.py/streaks.py を参照実装とし、公式 API の実レスポンスを Fixtures として埋め込み、assertion でフィールド一致・ヘッダ・キー選択(月替わり)を検証する。テストは TDD で先行させ、対象モジュールのカバレッジは最小 80% を目安とする。
NFR-09設定/秘密情報API キー・接続情報は環境変数で管理し、コードに埋め込まない。ゲスト用パスワード等は NEXT_PUBLIC_* の公開範囲を意識して扱う。
NFR-10UI/テーマnext-themes によるテーマ切替を維持。atomic design(atoms/molecules/organisms/pages)構成を保つ。

7. スコープ

移行を「即時 $0 でデプロイできる最小(MVP)」→「ユーザーデータ連携(段階2)」の二段で切る。

7.1 MVP(第 1 段: 最短デプロイ・即時 $0)

  • TVER 閲覧: ホーム(ランキング合成)・キーワード検索・シリーズ一覧・エピソード情報(FR-01〜FR-05)。
  • 再生: 純 TS のストリーム解決 + HLS 再生(FR-06,07)。Azure Functions/yt-dlp に依存しない。
  • 認証: Firebase Auth の登録/ログイン/ログアウト + 匿名ゲスト(30 秒制限)。ID トークン検証は jose(FR-08〜FR-11)。
  • 運用: /api/health(FR-16)。Firestore を必要としないため即時 $0 で成立。

7.2 段階2(第 2 段: Firestore 連携)

  • お気に入り・視聴履歴・プロフィール(写真含む)を Firestore client SDK + セキュリティルールで本人所有として復元(FR-12〜FR-14)。
  • 管理者機能(ユーザー検索・ロール更新・監査ログ)(FR-15)。
  • ライブ m3u 再生(パスワード制)の維持(FR-17)。

7.3 対象外・廃止

対象扱い理由(事実)
tvapp-bff(.NET7)移行対象外・廃止候補2024 停止のレガシー。
Platform-Stream-Loader(Azure Functions + yt-dlp)廃止(純 TS で置換)唯一の重い/課金源。解決フローを内製する。
Cosmos DB / /api/cosmosdb/videosMVP 除外・段階的に Firestore へダウンロード済み動画一覧の課金系。Workers 構成では外部依存を減らす。
firebase-admin(gRPC Firestore)置換(jose + client SDK)Workers ランタイム非互換のため。

8. 前提・制約

  • ランタイム/言語: Node v24 / npm 11、TypeScript strict、Next.js 14.2.15(App Router)。tsc --noEmit は通る(健全)。
  • テスト: 現状テスト FW 未導入。vitest を導入し、ストリーム解決を TDD で実装する。各段の期待値は公式 API 実レスポンスの Fixtures で固定し、参照実装(下記 yt-dlp)と突き合わせる(NFR-08)。
  • ブランチ: 作業は project/tvapp-cf-zero-cost 配下(例 feature/tvapp-streaks-resolver)。submodule は detached。
  • 外部依存の可変性 / 参照実装: TVER / Streaks / Brightcove の API 仕様・キー・可用性は外部要因で変わりうる(月替わりキー・ヘッダ要件など)。yt-dlp 2026.03.17 の tver.py/streaks.py を参照実装とし、月替わりキー選択・必須ヘッダ・レスポンス構造の検証基準とする。将来の drift(仕様変更)はこの参照実装との差分で追跡する。
  • 地域: TVER は JP 限定。日本 IP を前提とし、キー選択は JST 基準(NFR-04)。
  • 文書/コミット: ドキュメントは AGENTS-SHARED §8 準拠の自己完結 HTML。コミットは日本語・AI 署名なし(§2/§3)。本書は事実のみ(§10)。
