本書は tvapp-web の 要件定義とアーキテクチャを定める。記述は調査ブリーフおよび src/app/api / src/lib の実コードに基づく客観的事実に徹し、機能・非機能・外部依存・ データフロー・移行方針を扱う。TVER・Streaks・Firebase・Cloudflare の各 API 仕様は公開エンドポイントの 実測挙動を記述したものである。本書はリリース版を持たない策定文書であり、進行状況は上部の「ステータス」で示す。 「参考実装版」タグは本書作成時点の tvapp-web 実装版(1.4.0)を参考として併記したもので、本書自体の版ではない。 版上げの基準(要件・機能の変更 = MINOR、大構成の変更 = MAJOR)はプロジェクト直下の CLAUDE.md「リリースと版上げ」に従う。

1. 目的とスコープ

tvapp-web は TVER の公開 API を代理して番組メタデータ・ランキング・検索・エピソード情報を取得し、 HLS(m3u8)を react-player で再生する Web アプリである。認証は Firebase Auth を用いる。 本プロジェクト project/tvapp-cf-zero-cost の目的は、ストリーム解決を担う Azure Functions(yt-dlp)・Cosmos DB・firebase-admin への依存を除去し、 Cloudflare Workers(OpenNext)+ 純 TypeScript でのストリーム解決 + Firebase という構成へ移行して、 無料枠中心のほぼ $0 運用を成立させることにある。

1.1 スコープ内 / スコープ外

区分内容
スコープ内tvapp-web(Next.js 14 App Router / フロント + BFF)の要件・構成、TVER→Streaks の m3u8 解決を純 TS 化する設計、認証とユーザーデータの Workers 互換化
スコープ外(参照のみ)Platform-Stream-Loader(Azure Functions・yt-dlp)は移行後に廃止対象。tvapp-bff(.NET7・2024 停止)は移行対象外の廃止候補

3. システム構成

現行は Next.js の BFF から TVER 公開 API・Azure Functions・Cosmos DB・Firebase へ分岐する。 目標は Cloudflare Workers(OpenNext)上で BFF を動かし、m3u8 解決を純 TS で BFF 内に取り込み、 Azure Functions / Cosmos DB / firebase-admin を除去する。

6. データフロー(セッション 〜 m3u8 解決)

目標構成での再生解決フローを示す。すべての呼び出しは BFF(Workers)を起点とし、TVER・Streaks の公開エンドポイントへ 素の HTTP で到達する。セッション以外の TVER 呼び出しは Origin/Referer=tver.jp と x-tver-platform-type: web を付ける。

6.1 手順 6 の詳細(Playback 解決)

GET https://playback.api.streaks.jp/v1/projects/{projectID}/medias/ref:{videoRefID}
Accept: application/json
Origin:  https://tver.jp        # 必須。異なると 403(id139)
Referer: https://tver.jp/
X-Streaks-Api-Key: {key0{idx}}  # idx = (JST の月 % 6) || 6

→ sources[]: type=application/x-mpegURL の src(m3u8)。
   key_systems を持つソースは DRM のため除外し、非 DRM(clear)HLS を採用。
   字幕は tracks[](kind = captions / subtitles)。
   旧作で streaks が無い場合は Brightcove(accountID / videoRefID + playerID)へフォールバック。
TVER は JP 限定(geo)。日本 IP から Playback 解決は HTTP 200 / sources=1 / 非 DRM HLS を実測。 この経路は yt-dlp 2026.03.17 の tver.py + streaks.py の挙動と一致する。 Streaks の 403 応答に含まれる id は実測値であり、src/lib/tver/streamResolver.test.ts で id=124(geo)/ id=126(key)/ id=139(視聴不許可)を再現している。

7. 非機能要件

7.1 性能

項目要件・事実
解決レイテンシ再生解決は 手順 3→4→5→6 が直列依存(version → 静的 JSON → キー → Playback)。手順 2 のランキングはジャンル 5 本を Promise.all で並列化済み
キャッシュstreaks_info(キー)は月替わりのため Workers KV または短期メモリでキャッシュし、キー選択は各リクエスト時の JST 月から算出
ペイロードBFF は upstream の JSON を中継。call/home のみランキングを合成して返す

7.2 コスト

項目要件・事実
ホスティングCloudflare Workers 無料枠 10 万 req/日。@opennextjs/cloudflare + nodejs_compat
ストリーム解決純 TS で BFF 内に取り込み、Azure Functions / yt-dlp / Cosmos DB の課金源を全廃
MVP の $0 化TVER 閲覧 + 再生 + Firebase Auth のみなら Firestore 不要で即時 $0。お気に入り / 履歴 / プロフィール / 管理は段階 2

7.3 セキュリティ

項目要件・事実
upstream ヘッダTVER 代理は Origin/Referer(session=s.tver.jp、他=tver.jp)と x-tver-platform-type: web、Streaks は Origin: https://tver.jp + X-Streaks-Api-Key を付与
認証(現行)ユーザー系は firebase-admin の verifyIdToken、内部 JWT は jose の jwtVerify(JWT_SECRET_KEY)
認証(目標)firebase-admin(gRPC Firestore は Workers 非互換)を廃し、jose + Google 公開鍵で Firebase ID トークンを検証(Workers 互換・依存軽量)
データ所有段階 2 で Firestore client SDK + セキュリティルールにより本人所有データ(users/{uid}/...)を直接読み書き
秘密情報Firebase 秘密鍵・Cosmos Key・Azure Code Key・m3u URL は環境変数。ソースにハードコードしない
アクセス制御匿名(ゲスト)ユーザーは一部リソースで 403。エピソード再生は 30 秒制限

7.4 可用性

項目要件・事実
地理制約TVER は JP 限定(geo)。日本 IP からの疎通を前提とする
外部 API 依存再生は TVER + Streaks の公開 API に依存。call/home は個別ジャンルの失敗を null で許容し、取得できたジャンルのみ合成する
ランタイム互換Workers では Node 固有 API を nodejs_compat で吸収。gRPC 依存(firebase-admin Firestore)は不採用
ヘルスチェック/api/health で稼働と環境変数の設定有無を確認

8. 移行方針

移行は 2 段階で行う。段階 1 は Firestore 不要で $0 を成立させ、段階 2 でユーザーデータを Workers 互換に置換する。 「何を変えるか」は §8.3 の対応表に集約し、§8.1・§8.2 は各段階の実装手順(順序)に絞る(責務を分ける)。

8.1 段階 1 の進め方(実装手順・$0 の最短デプロイ)

  1. ホスティングを Cloudflare Workers(OpenNext・nodejs_compat)へ移す(以降の土台)。
  2. 純 TS のストリーム解決(session→episode→statics→streaks_info→playback。図2)を実装し、streaminglink の Azure Functions 依存を差し替える。
  3. streaks_info を Workers KV(またはメモリ + 短期)でキャッシュする。
  4. Firebase Auth(client SDK・email/password + 匿名ゲスト)で最短デプロイ。提供機能は TVER 閲覧(ランキング / 検索 / シリーズ / エピソード)+ 再生 + ログイン / 登録 / ゲスト(Firestore 不要)。

8.2 段階 2 の進め方(実装手順・ユーザーデータの Workers 互換化)

  1. サーバの ID トークン検証を firebase-admin から jose + Google 公開鍵へ置換する。
  2. お気に入り / 視聴履歴 / プロフィールを Firestore client SDK + セキュリティルールで本人所有データとして読み書きする。
  3. Cosmos DB(cosmosdb/videos)を Firestore へ集約、または廃止を判断する。
  4. Azure Functions(Platform-Stream-Loader)を廃止し、tvapp-bff(.NET7・停止中)を廃止候補として整理する。
